# Creating Skills

## Чеклист

- [ ] 1. Изучить источник (лог чата, архивная задача, опыт пользователя).
- [ ] 2. Вычленить **только проектно-специфичную** логику. Убрать всё, что агент знает сам.
- [ ] 3. Сгенерировать `name` (gerund, kebab-case, макс. 64 символа).
- [ ] 4. Написать `description` (WHAT + WHEN, 3-е лицо, макс. 1024 символа).
- [ ] 5. Оформить тело: примеры > проза, шаблоны > описания.
- [ ] 6. Проверить по финальному чеклисту (см. ниже).
- [ ] 7. Записать через `write_to_file` в `.agents/skills/<name>/SKILL.md`.

## Структура файла

```
.agents/skills/<skill-name>/
├── SKILL.md          (обязательно)
├── references/       (опционально — справочные данные, если >500 строк)
└── examples/         (опционально — полные примеры кода)
```

## YAML Frontmatter

```yaml
---
name: processing-save-data
description: Обрабатывает сериализацию и десериализацию сохранений через Newtonsoft.Json. Вызывается при создании или модификации системы сохранений в проекте.
---
```

**Правила:**
- `name`: только `[a-z0-9-]`, форма gerund (`managing-X`, `creating-Y`).
- `description`: одно предложение ЧТО + одно предложение КОГДА.

## Принципы написания тела

### 1. Примеры > Проза

```
❌ ПЛОХО (проза):
"TemplateContainer — это скрытый узел, который Unity создаёт
автоматически при использовании ui:Instance. Он является
обычным flex-элементом и участвует в раскладке родителя..."

✅ ХОРОШО (пример вход → выход):

<!-- НЕПРАВИЛЬНО — класс на внутреннем элементе -->
<ui:Instance template="Panel" />
<!-- Panel.uxml: -->
<ui:VisualElement class="my-absolute-panel">  ← не сработает

<!-- ПРАВИЛЬНО — класс на Instance -->
<ui:Instance template="Panel" class="my-absolute-panel" />
```

### 2. Focus on What's Missing

Не объясняй стандартные концепции (что такое `position: absolute`, как работает flex). Описывай **только** то, что отличается от ожиданий агента — ловушки, проектные соглашения, нестандартное поведение.

### 3. Таблицы для справочников

Используй таблицы для маппингов «запрещено → замена», «свойство → значение»:

```markdown
| Запрещено | Замена |
|---|---|
| `gap` | `margin-bottom` на дочерних |
| `right` | `left` с абсолютной координатой |
```

### 4. Шаблоны для выходных форматов

Если скилл генерирует код — давай готовый шаблон, который агент копирует и заполняет:

```xml
<!-- Шаблон UXML -->
<ui:Style src="../USS/Variables.uss" />
<ui:Style src="../USS/<Component>.uss" />
<ui:VisualElement name="<root-name>">
    <!-- контент -->
</ui:VisualElement>
```

### 5. Progressive Disclosure

- Тело SKILL.md ≤ 500 строк.
- Если справочные данные длинные — выноси в `references/` с инструкцией, когда читать: *«См. `references/error-codes.md` при обработке ошибок API.»*

## Шаблон SKILL.md

```markdown
---
name: <gerund-kebab-case>
description: <ЧТО делает>. <КОГДА вызывается>.
---

# <Название>

## Чеклист

- [ ] 1. Шаг
- [ ] 2. Шаг

## Быстрый старт

<Минимальный пример входа → выхода>

## Ловушки

<Таблица или список конкретных ошибок с примерами до/после>

## Шаблоны

<Готовые шаблоны кода для копирования>

## Ресурсы

- <Ссылки на файлы проекта>
```

## Финальный чеклист

- [ ] Frontmatter: `name` (gerund, kebab) + `description` (WHAT + WHEN, 3-е лицо)
- [ ] Тело < 500 строк
- [ ] Нет объяснений стандартных концепций (агент их знает)
- [ ] Каждая ловушка проиллюстрирована примером ❌/✅
- [ ] Есть готовые шаблоны кода для копирования
- [ ] Пути только через `/`, не `\`
