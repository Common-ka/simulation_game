# Мастер планирования архитектуры (Project Roadmap Planning)

Этот скилл предназначен для формирования пула задач. В отличие от Промптера (который переводит одно ТЗ в задачу для Воркера), Плэннер должен сформировать **серию взаимосвязанных ТЗ**, объединяющихся в один Милстоун.

## Шаг 1: Правило "Шестеренки" (Связи систем)
Никогда не планируй фичу в вакууме. При планировании (например, `GachaController`), всегда проверяй:
- Хранение: Записывается ли это в `RetainedData` (`SaveManager.cs`)?
- Зависимости: Использует ли она конфиги загруженные через `GameDataLoader`?
- Интерфейсы: Нужно ли пробрасывать события (Events) для UI Toolkit?
Если зависимостей много, выноси их реализацию в **отдельные** смежные задачи.

## Шаг 2: Алгоритм нарезки Промптов (Micro-Tasking)
Идеально нарезанный план (implementation_plan.md) выглядит так:

1. **Фаза данных:** "Промптеру: Создать модели JSON и обновить парсинг в DataLoader".
2. **Фаза логики ядра:** "Промптеру: Создать Controller, написать логику без UI и эффектов, добавить Event-выброс."
3. **Фаза сохранения:** "Промптеру: Обновить SaveModels, добавить подписку в SaveManager."
4. **Фаза визуализации (если доступно):** "Промптеру: Создать USS/UXML и скрипт UIController, подписав его на Controller."

## Шаг 3: Техника безопасности ИИ
- **Не планируй рефакторинг "на лету"**: Если старый код плох, создавай под это отдельную таску с пометкой `[Refactor]`, а не заставляй Воркера пилить фичу и рефакторить одновременно.
- **Требуй индексации**: В каждом промпте для Промптера напоминай: *"Воркер должен обновить LTM запись в INDEX.md"*. Это спасет долговременную память системы.

Используй этот скилл как чеклист перед тем, как отдать пользователю готовый `implementation_plan.md`.
